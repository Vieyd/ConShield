# Черновик разделов ВКР по ConShield

## Тема

Разработка комплекса защиты контейнерных приложений на основе сканирования образов, политик выполнения и мониторинга runtime-событий.

## Объект и предмет

Объектом исследования является контейнеризированное приложение и его жизненный цикл: подготовка образа, запуск контейнера, выполнение workload, сбор событий безопасности и последующее реагирование оператора.

Предметом исследования являются методы DevSecOps-защиты контейнерных приложений, основанные на сканировании образов, проверке политики запуска, мониторинге runtime-событий, обработке событий безопасности и SIEM-корреляции.

## Цель работы

Цель работы — разработать и проверить прототип комплекса ConShield, который объединяет сканирование контейнерных образов, локальную политику запуска, сбор runtime-событий, защищённую доставку событий, SIEM-корреляцию и операторские интерфейсы для анализа состояния защищаемой среды.

## Задачи работы

1. Проанализировать угрозы безопасности контейнерных приложений.
2. Спроектировать архитектуру комплекса защиты.
3. Реализовать модуль сканирования контейнерных образов.
4. Реализовать policy gate для проверки решений запуска контейнеров.
5. Реализовать приём и мониторинг runtime-событий от сенсора.
6. Реализовать обработку событий и SIEM-корреляцию.
7. Реализовать жизненный цикл сенсоров и управление credential-данными.
8. Реализовать операторский Web UI и отчётность.
9. Провести валидацию прототипа локальными тестами и безопасным демонстрационным сценарием.

## Актуальность

Контейнеризация ускоряет разработку и развёртывание приложений, но одновременно переносит часть рисков в цепочку поставки образов, настройки запуска и runtime-поведение контейнера. Уязвимые образы, небезопасные параметры запуска, подозрительные процессы внутри контейнера и недостаточная наблюдаемость могут привести к компрометации приложения или инфраструктуры.

Для снижения этих рисков важны несколько связанных контуров: проверка образа до запуска, применение политики выполнения, мониторинг runtime-событий, защищённая доставка событий в центральное хранилище и корреляция событий в SIEM-подобной системе. ConShield реализует такой подход как учебный прототип, пригодный для демонстрации архитектурных решений и операторского workflow.

## Архитектура решения

Архитектура ConShield построена как последовательная цепочка: сканирование образа → проверка политики → runtime-мониторинг → защищённый ingestion API → PostgreSQL → outbox/RabbitMQ/EventConsumer → SIEM-корреляция → Alerts/Incidents → Web UI и отчёты.

Модуль ImageScanner использует Trivy для анализа контейнерных образов и преобразует результаты в события безопасности. Policy gate оценивает локальную политику запуска и фиксирует решение Allow/Warn/Block. RuntimeCollector принимает Falco-compatible события и передаёт их в центральный ConShield Web через защищённый ingestion API.

PostgreSQL используется как основное хранилище событий, сенсоров, credential-verifier записей, alert/incidents и outbox/inbox данных. Outbox, RabbitMQ и EventConsumer обеспечивают более надёжную доставку и обработку событий. SIEM-правила создают alerts и incidents на основе повторяющихся или значимых событий. Web UI предоставляет страницы Security Events, Sensor Fleet, Operations Health и Security Summary.

## Реализованные модули

### Сканирование образов

Модуль `ConShield.ImageScanner` запускает проверку контейнерного образа и может работать в безопасном режиме без отправки события. Результаты сканирования используются как evidence для анализа риска образа и могут участвовать в SIEM-корреляции.

### Policy gate

Модуль `ConShield.ContainerPolicy` реализует локальную проверку политики запуска контейнера. Он позволяет показать, как решение запуска может быть формализовано в виде Allow/Warn/Block и сохранено как событие безопасности.

### Runtime collector

Модуль `ConShield.RuntimeCollector` обрабатывает Falco-compatible JSONL события, нормализует их и отправляет в ConShield как runtime-события от enrolled sensor. Такой подход демонстрирует мониторинг поведения контейнера после запуска.

### Ingestion API и хранилище

ConShield Web содержит защищённый API приёма событий. События сохраняются в PostgreSQL, а дальнейшая доставка поддерживается transactional outbox и RabbitMQ/EventConsumer pipeline.

### SIEM и operator workflow

SIEM-корреляция создаёт alerts/incidents по реализованным правилам, включая lifecycle SIEM правила `LIFE-001` и `LIFE-002`. Оператор использует Security Events, Sensor Fleet, Operations Health и Security Summary для анализа состояния системы.

### Sensor lifecycle и отчётность

Система содержит sensor inventory, heartbeat, provisioning, rotation/revocation workflow и lifecycle audit events. Security Summary report/export формирует агрегированную read-only сводку без вывода секретов и raw payload данных.

## Безопасность и ограничения

Прототип не хранит plaintext credential-данные сенсоров. Для sensor credentials используются verifier-значения, которые не должны выводиться оператору или попадать в отчёты. Разрушительные lifecycle-действия ограничены ролью AdminIB. Runtime-события резервного source system принимаются по sensor-bound authentication, legacy fallback остаётся отключённым.

Отчётность спроектирована как secret-safe: в Markdown export не включаются raw `AdditionalDataJson`, API keys, connection strings, env values, cookies, tokens, passwords или локальные секреты. При демонстрации нельзя показывать generated credentials, `appsettings.Development.json`, Fedora env files, RabbitMQ/PostgreSQL passwords и любые screenshots/logs с секретами.

ConShield является учебным прототипом. Он не является промышленно готовым решением и не должен рассматриваться как самостоятельная достаточная защита. Для промышленного применения требуются дополнительные меры: production identity, централизованное управление секретами, mTLS, observability, retention policy и эксплуатационное hardening.

## Сценарий демонстрации

Безопасный сценарий демонстрации начинается с запуска локальной инфраструктуры и входа под пользователем AdminIB. Далее показываются Operations Health, Sensor Fleet, Security Events и фильтры lifecycle audit events. После этого можно показать SIEM alerts/incidents, открыть Security Summary report и объяснить, что Markdown export содержит только агрегированные counts/timestamps.

Дополнительно можно показать CLI-команды ImageScanner и PolicyGate в no-submit режиме, а также состояние Fedora runtime sensor только на уровне service status и permission checks. Содержимое protected env files и generated credentials не демонстрируется.

## Результаты

В результате работы создан прототип комплекса защиты контейнерных приложений, который связывает несколько уровней контроля: анализ образа, policy gate, runtime-события, защищённый ingestion, событийное хранилище, outbox/RabbitMQ/EventConsumer pipeline, SIEM-корреляцию, sensor lifecycle и операторскую отчётность.

Проект демонстрирует полный explainable security workflow для учебной защиты: от события на уровне контейнера до расследования оператором и формирования безопасной summary-сводки.

## Направления дальнейшего развития

- mTLS для усиления sensor/service identity.
- Kubernetes admission controller для проверки workload до запуска в кластере.
- Централизованный secret manager для production credential lifecycle.
- Metrics/observability через Prometheus/Grafana.
- Configurable rules или rule DSL для управления SIEM-корреляцией без перекомпиляции.
- Retention/partitioning для долговременного хранения событий.
- Production hardening authentication, authorization, deployment и service isolation.
