# Attacker Scenarios

These scenarios define the threat paths used by ConShield requirements and traceability. They are intentionally scoped to the current local-first implementation.

## AS-001 — Critical vulnerability in container image reaches CI/CD

- Attacker goal: get a vulnerable image promoted through a pipeline.
- Preconditions: an image scan result exists or can be supplied as a fixture.
- Attack path: vulnerable image is scanned; findings are ignored or not connected to a decision.
- ConShield detection/prevention path: image scan normalization produces an IMG event; configurable SIEM rules can raise `IMG-001`; CI/CD gate evaluates policy and returns deterministic exit behavior.
- Relevant modules: `ConShield.ImageScanner`, `ConShield.ContainerPolicy`, `ConShield.Cli`, `ConShield.Application`.
- Relevant rules/configs: `IMG-001`, `config/container-policy.default.json`, `config/siem-rules.default.json`.
- Relevant CLI/demo command: `dotnet run --project .\src\ConShield.Cli -- gate image --from-trivy-json .\tests\TestData\Trivy\sample-image-scan.json --fail-on block --no-submit`.
- Evidence produced: Image Scan Evidence, Container Policy Evidence, SIEM Rules Evidence, CI/CD gate report.
- Residual risk: fixture validation does not prove live scanner database freshness.

## AS-002 — Operator attempts protected run of blocked image

- Attacker goal: start a container even when policy says it should be blocked.
- Preconditions: protected run command receives scan findings that match a block rule.
- Attack path: operator or automation attempts to continue despite high-risk findings.
- ConShield detection/prevention path: protected run evaluates policy, records a POL decision, and does not run blocked containers.
- Relevant modules: `scripts/Invoke-ConShieldProtectedRun.ps1`, `ConShield.ContainerPolicy`, `ConShield.Cli`.
- Relevant rules/configs: `POL-001`, `config/container-policy.default.json`.
- Relevant CLI/demo command: `dotnet run --project .\src\ConShield.Cli -- run protected --from-trivy-json .\tests\TestData\Trivy\sample-image-scan.json --no-run --no-submit`.
- Evidence produced: Protected Run Evidence, Container Policy Evidence, Operator Workflow.
- Residual risk: live Docker execution remains an explicit manual path and depends on host controls.

## AS-003 — Runtime shell spawned inside container

- Attacker goal: execute an interactive shell or suspicious process inside a running container.
- Preconditions: runtime event source or deterministic Falco-compatible fixture exists.
- Attack path: shell execution event is generated by runtime monitoring.
- ConShield detection/prevention path: runtime mapper normalizes the event into an RTE security event and SIEM rules raise `RTE-001`.
- Relevant modules: `ConShield.RuntimeDetection`, `ConShield.RuntimeCollector`, `scripts/Replay-ConShieldFalcoRuntimeEvent.ps1`.
- Relevant rules/configs: `RTE-001`, `config/siem-rules.default.json`.
- Relevant CLI/demo command: `dotnet run --project .\src\ConShield.Cli -- sensor replay --demo-signature --no-submit`.
- Evidence produced: Runtime Sensor Health, Signed Sensor Event Evidence, RTE alert/incident evidence.
- Residual risk: local replay does not replace host-level runtime protection.

## AS-004 — Abnormal Docker lifecycle event indicates suspicious behavior

- Attacker goal: hide or exploit abnormal container lifecycle behavior.
- Preconditions: Docker-compatible lifecycle fixture or live event source exists.
- Attack path: container starts, stops, dies, or is destroyed in a suspicious sequence.
- ConShield detection/prevention path: lifecycle collector normalizes events into LIFE security events and SIEM rules raise `LIFE-001` or `LIFE-002`.
- Relevant modules: `ConShield.Cli`, Docker lifecycle collector workflow, `ConShield.Application`.
- Relevant rules/configs: `LIFE-001`, `LIFE-002`, `config/siem-rules.default.json`.
- Relevant CLI/demo command: `dotnet run --project .\src\ConShield.Cli -- lifecycle replay --from-docker-events-json .\tests\TestData\DockerEvents\container-lifecycle-events.json --no-submit`.
- Evidence produced: Docker Lifecycle Collector Evidence, SIEM alert evidence.
- Residual risk: live Docker event watching is optional and host-dependent.

## AS-005 — Unknown runtime sensor submits event

- Attacker goal: submit runtime data from an unregistered source.
- Preconditions: event includes a runtime source not trusted in the default registry.
- Attack path: unknown source attempts to blend into normal runtime telemetry.
- ConShield detection/prevention path: sensor registry marks source as Unknown and enforcement raises `SENSOR-001`.
- Relevant modules: sensor trust registry, runtime sensor health, SIEM correlation.
- Relevant rules/configs: `SENSOR-001`, `config/sensor-registry.default.json`, `config/siem-rules.default.json`.
- Relevant CLI/demo command: `pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Replay-ConShieldFalcoRuntimeEvent.ps1 -SimulateUnknownSensor -NoSubmit`.
- Evidence produced: Sensor Trust Evidence, Sensor Trust Enforcement Evidence, Runtime Sensor Health.
- Residual risk: unknown-source handling is not full cryptographic device identity.

## AS-006 — Revoked or disabled sensor submits event

- Attacker goal: continue using a source that should no longer be trusted.
- Preconditions: source status is Revoked or Disabled in the sensor registry.
- Attack path: revoked/disabled source submits runtime event metadata.
- ConShield detection/prevention path: trust enforcement raises stricter handling through `SENSOR-002`.
- Relevant modules: sensor trust registry, trust enforcement, runtime sensor health.
- Relevant rules/configs: `SENSOR-002`, `config/sensor-registry.default.json`, `config/siem-rules.default.json`.
- Relevant CLI/demo command: `pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Replay-ConShieldFalcoRuntimeEvent.ps1 -SimulateRevokedSensor -NoSubmit`.
- Evidence produced: Sensor Trust Enforcement Evidence, SIEM alert/incident evidence.
- Residual risk: future production deployments need stronger key and credential lifecycle controls.

## AS-007 — Missing/invalid/stale signature on runtime sensor event

- Attacker goal: submit forged or replay-like runtime event metadata.
- Preconditions: runtime sensor envelope lacks a valid signature state.
- Attack path: event is missing signature metadata, has invalid signature metadata, or appears stale.
- ConShield detection/prevention path: signed sensor event verifier produces deterministic signature status and SIEM raises `SIGN-001`, `SIGN-002`, or `SIGN-003`.
- Relevant modules: signed sensor event verifier, runtime mapper, Runtime Sensor Health.
- Relevant rules/configs: `SIGN-001`, `SIGN-002`, `SIGN-003`, `config/siem-rules.default.json`.
- Relevant CLI/demo command: `pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Replay-ConShieldFalcoRuntimeEvent.ps1 -SimulateInvalidSignature -NoSubmit`.
- Evidence produced: Signed Sensor Event Evidence, Runtime Sensor Health.
- Residual risk: demo signing material is not production key management.

## AS-008 — Replayed or duplicate external event

- Attacker goal: create alert noise or hide identity by resubmitting an existing event.
- Preconditions: external event identity is repeated.
- Attack path: duplicate event is submitted through ingestion or replay script.
- ConShield detection/prevention path: deterministic external IDs and ingestion deduplication prevent duplicate rows or surface duplicate counts.
- Relevant modules: external event ingestion, image scan/protected run/runtime replay scripts, tests.
- Relevant rules/configs: deterministic external event ID conventions.
- Relevant CLI/demo command: fixture commands with `--no-submit` for deterministic ID validation, or local ingestion smoke when services are available.
- Evidence produced: Outbox/inbox summary, event counts, duplicate counters.
- Residual risk: distributed replay detection across multiple production nodes is future work.

## AS-009 — Unsafe evidence/report generation could leak raw payloads

- Attacker goal: expose sensitive raw payloads, local overrides, logs, or secret-like material through generated reports.
- Preconditions: evidence/export/report command reads application state.
- Attack path: raw fields are accidentally copied into Markdown or release artifacts.
- ConShield detection/prevention path: evidence exporter and release pack use safe summaries and tests guard against raw payload markers.
- Relevant modules: `scripts/Export-ConShieldDefenseEvidence.ps1`, `scripts/New-ConShieldDemoReleasePack.ps1`, docs/tests.
- Relevant rules/configs: `.gitignore`, release pack allowlist.
- Relevant CLI/demo command: `pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Export-ConShieldDefenseEvidence.ps1 -OutputMarkdownPath .\artifacts\local\defense-evidence.md`.
- Evidence produced: sanitized defense evidence Markdown under ignored local artifacts.
- Residual risk: operators must still avoid placing real secrets into demo input.

## AS-010 — Misconfigured SIEM or policy config weakens detection

- Attacker goal: reduce alerting or weaken launch decisions through config mistakes.
- Preconditions: SIEM or policy config changes.
- Attack path: required rule IDs, thresholds, or policy decisions are accidentally changed.
- ConShield detection/prevention path: config validation scripts and CI-safe tests verify required rule/config contracts.
- Relevant modules: SIEM rules validation, container policy validation, full validation wrapper.
- Relevant rules/configs: `config/siem-rules.default.json`, `config/container-policy.default.json`.
- Relevant CLI/demo command: `pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Test-ConShieldFullValidation.ps1`.
- Evidence produced: full validation PASS output, SIEM Rules Evidence, Container Policy Evidence.
- Residual risk: production governance would need approval workflow and environment separation.
