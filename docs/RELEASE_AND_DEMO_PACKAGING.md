# ConShield Release and Demo Packaging

Product Packaging & Demo Release Pack v1 creates a local, gitignored handoff bundle for diploma/demo use. It is not an installer and does not add a new security subsystem. It packages the current safe ConShield workflows into one local folder and archive under `artifacts/local`.

## Create the pack

Run from the repository root:

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\New-ConShieldDemoReleasePack.ps1
```

Expected high-level output:

```text
ConShield demo release pack
Restore: OK
Build: OK
Test: OK
Full validation: OK
CLI publish: OK
Docs index: OK
Release pack: artifacts/local/conshield-demo-release-pack
Archive: artifacts/local/conshield-demo-release-pack.zip
Result: PASS
```

Useful switches:

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\New-ConShieldDemoReleasePack.ps1 -SkipTests
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\New-ConShieldDemoReleasePack.ps1 -SkipFullValidation
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\New-ConShieldDemoReleasePack.ps1 -NoArchive
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\New-ConShieldDemoReleasePack.ps1 -IncludeEvidenceTemplate
```

Default packaging runs restore, build, tests, full validation, CLI publish, safe file copy, pack safety checks, and archive creation.

## Output layout

Default output:

```text
artifacts/local/conshield-demo-release-pack/
  README.md
  bin/conshield-cli/
  config/
  docs/
  scripts/

artifacts/local/conshield-demo-release-pack.zip
artifacts/local/conshieldctl/
```

`artifacts/local/` is ignored by Git. The release folder, archive, and published CLI output must remain local.

## Intentionally excluded

The pack must not include:

- secrets, API keys, passwords, tokens, cookies, connection strings, or environment values;
- `.env` files;
- local overrides such as `appsettings.Development.json`, `appsettings.Local.json`, or config `*.local.json`;
- generated evidence, generated reports, logs, screenshots, raw scanner/runtime payloads, raw `AdditionalDataJson`, raw Docker event JSON, or raw Trivy JSON;
- nested `artifacts/local` content;
- source `.git`, root `bin/obj`, `TestResults`, database files, or local Docker/Falco runtime artifacts.

The packaging script refuses generated/local paths and runs a path safety check before creating the archive.

## Validate before packaging

The default packaging command runs full validation. You can also run it explicitly:

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Test-ConShieldFullValidation.ps1
```

This default validation path is deterministic and does not require live Web/API, live Docker execution, live Trivy DB/network, real Fedora/Falco, external internet, browser login, real certificates, private keys, signing keys, or real secrets.

## Use the packed CLI

After packaging:

```powershell
dotnet .\artifacts\local\conshield-demo-release-pack\bin\conshield-cli\ConShield.Cli.dll --help
dotnet .\artifacts\local\conshield-demo-release-pack\bin\conshield-cli\ConShield.Cli.dll validate
```

The packed CLI can show command help and run deterministic validation commands. Workflows that need repository scripts, fixtures, local services, or source files should still be run from the repository root.

## Product positioning docs

The release pack includes concise defense and positioning docs:

- [PRODUCT_POSITIONING.md](PRODUCT_POSITIONING.md) explains ConShield as a local-first DevSecOps/SIEM control plane for container security.
- [COMPETITIVE_ANALYSIS.md](COMPETITIVE_ANALYSIS.md) compares ConShield cautiously with scanner-focused, runtime-focused, Kubernetes policy, SIEM/XDR, and commercial CNAPP categories.
- [DIPLOMA_DEFENSE_NARRATIVE.md](DIPLOMA_DEFENSE_NARRATIVE.md) provides defense wording for the scan → policy gate → runtime/lifecycle → sensor trust/signatures → SIEM → incidents → evidence story.
- [ROADMAP_TO_PRODUCTION.md](ROADMAP_TO_PRODUCTION.md) lists realistic hardening work without claiming the local prototype already covers enterprise production scope.

The pack also includes [THREAT_MODEL.md](THREAT_MODEL.md), [ATTACKER_SCENARIOS.md](ATTACKER_SCENARIOS.md), [SECURITY_REQUIREMENTS.md](SECURITY_REQUIREMENTS.md), [REQUIREMENTS_TRACEABILITY_MATRIX.md](REQUIREMENTS_TRACEABILITY_MATRIX.md), and [RESIDUAL_RISKS.md](RESIDUAL_RISKS.md) so a reviewer can trace threats to requirements, modules, commands, SIEM rules, evidence, and tests.

Architecture and data-flow explanation are included through [ARCHITECTURE.md](ARCHITECTURE.md), [ARCHITECTURE_DIAGRAMS.md](ARCHITECTURE_DIAGRAMS.md), [DATA_FLOW_MODEL.md](DATA_FLOW_MODEL.md), [DEPLOYMENT_VIEW.md](DEPLOYMENT_VIEW.md), and [SEQUENCE_FLOWS.md](SEQUENCE_FLOWS.md). The diagrams are Mermaid in Markdown, not generated binary images.

## Operator dashboard

The Web UI includes a read-only operator dashboard at:

```text
http://127.0.0.1:5080/Dashboard
```

The dashboard is useful during a demo because it shows posture cards, latest sanitized alerts/incidents, sensor trust/signature summaries, workflow tiles, command snippets, and docs links in one place. It does not run local commands from the browser and does not display secrets, raw payloads, logs, environment values, connection strings, or generated local artifacts.

## Start Web GUI for demo

Run from the repository root:

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File .\Start-ConShield.ps1 -StartApps -OpenRabbit
```

Then open:

```text
http://127.0.0.1:5080/Demo
```

The `/Demo` page lists the demo order, safe commands, links to Security Summary, Security Events, SIEM, Incidents, Runtime Sensor Health, and the packaging command. It does not execute shell scripts from the browser.

## Export evidence

Run from the repository root after local services are available:

```powershell
pwsh -NoProfile -ExecutionPolicy Bypass -File .\scripts\Export-ConShieldDefenseEvidence.ps1 `
  -OutputMarkdownPath .\artifacts\local\defense-evidence.md
```

Generated evidence remains local and is not included in the pack by default. Use `-IncludeEvidenceTemplate` only to add an empty instruction placeholder.

## Optional checks

These remain intentionally outside the release pack:

- real Fedora/Falco deployment;
- live Docker execution;
- live Trivy DB/network scanning;
- external internet-dependent runtime checks;
- Kubernetes/admission controller;
- full mTLS;
- real certificates, private keys, or signing keys;
- production secret rotation.

Use the dedicated operational docs for those checks.
